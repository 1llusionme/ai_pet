# PRD 评审报告 - Cogni-Pets (灵犀伙伴)
**评审对象**: `docs/PRD_CogniPets.md` (v2.0)
**评审人**: 产品总监 (Product Director) & 技术总监 (Technical Director)
**日期**: 2026-03-15

---

## 第一部分：产品总监评审 (Product Review)
**核心观点：概念性感，但落地细节存在“理想化”倾向，部分交互设计有高估用户（家长/儿童）耐心的嫌疑。**

### 1. 致命问题 (Critical Issues)
*   **家长端交互过于繁琐 (High Friction for Parents)**
    *   **原文引用**: "家长输入文本：'这周让他多练练英语口语' -> LLM 转化..."
    *   **批判**: 这是典型的“PM 自嗨”。家长都很忙/懒，没空跟 AI 聊天发布任务。
    *   **建议**: 必须提供 **Quick Actions (快捷指令)** 按钮库（如：[一键催作业]、[一键练口语]、[一键夸奖]）。文本输入只能作为高级选项，绝不能是主交互。
*   **费曼模式的“挫败感”风险 (Frustration Risk)**
    *   **场景**: AI 故意装傻。
    *   **盲点**: 4-6 岁儿童可能无法理解“反讽”或“装傻”。如果 AI 一直说“我不懂”，孩子会觉得“这个 AI 好笨”然后弃用，甚至发脾气。
    *   **建议**: 必须加入 **"Smart Fallback" (智能兜底)**。如果孩子解释了两次 AI 还在装傻，AI 必须立刻“强行听懂”并给予鼓励，绝对不能让孩子产生无力感。
*   **进化系统的深度不足 (Lack of Depth)**
    *   **批判**: 目前进化只改变了“皮囊” (Form)，没改变“灵魂” (Gameplay)。齿轮恐龙和云端独角兽如果玩起来一样，那这就只是个换皮游戏。
    *   **建议**: 进化必须解锁**新能力**。例如：恐龙形态解锁“数学大挑战”小游戏；独角兽形态解锁“绘本共读”功能。

### 2. 缺失模块 (Missing Modules)
*   **Onboarding (新手引导)**: 文档完全没提孩子第一次打开 App 怎么上手。对于儿童产品，Onboarding 是生与死的界限。
*   **Anti-Addiction (防沉迷)**: 既然是护眼/伴学，如果孩子沉迷于喂宠物不睡觉怎么办？需要强制休息机制。

## 第二部分：技术总监评审 (Technical Review)
**核心观点：硬件性能预估过于乐观，架构存在“算力爆炸”风险，数据结构支撑不了“真智能”。**

### 1. 致命风险 (Critical Risks)
*   **功耗与发热 (Thermal Throttling)**
    *   **原文引用**: "Unity 3D + Local LLM + Vision (5fps) + ASR"
    *   **批判**: 在 iPad A12 (2018) 上同时跑这些？**10分钟内必降频卡顿，半小时没电**。
    *   **修正方案**:
        *   **互斥机制**: 视觉识别时暂停 LLM 推理；LLM 推理时暂停 3D 渲染（降为静态图）。
        *   **Vision 降频**: 5fps 没必要，`1fps` 或 `0.5fps` 足以完成“找红色物体”的任务。
*   **延迟指标不切实际 (Unrealistic Latency)**
    *   **原文引用**: "ASR -> LLM -> TTS < 1.5s"
    *   **现实**: 本地 ASR (0.5s) + LLM Token生成 (首字 0.5s + 生成完 2s) + TTS (0.5s)。全链路至少 **3-5秒**。
    *   **建议**: 必须设计 **"Thinking Animation" (思考动画)** 来掩盖延迟（比如宠物歪头、挠耳朵），不能让用户干等。
*   **ASR 儿童语音识别率 (ASR Accuracy)**
    *   **痛点**: Whisper 对中文儿童（口齿不清、逻辑混乱）的识别率惨不忍睹。
    *   **对策**: 不能强依赖“关键词匹配”。必须结合 **Context (当前题目上下文)** 做 Intent Matching，而不是做全文本识别。

### 2. 架构缺陷 (Architecture Flaws)
*   **记忆系统过于简单 (Memory Too Simple)**
    *   **批判**: 仅用 `SQLite Key-Value` 存事实（UserFacts）是不够的。
    *   **场景**: 孩子说“我不喜欢吃红色的水果”。如果是 KV，你可能存 `dislike: red fruit`。下次孩子问“我有苹果你吃吗？”，KV 很难推断出苹果是红色的所以不吃。
    *   **建议**: 必须引入 **Vector Database (轻量级向量库，如 Chroma/FAISS-lite)**，配合 Embedding 实现语义级记忆检索。

## 第三部分：汇总改进建议 (Actionable Feedback)

| 优先级 | 模块 | 改进项 | 负责人 |
| :--- | :--- | :--- | :--- |
| **P0 (阻塞)** | **家长端** | 废除纯文本指令，改为 **"Dashboard + 快捷卡片"** 交互。 | 产品 |

| **P0 (阻塞)** | **延迟** | 设计 **"Masking Interaction" (掩盖式交互)** 动画，降低用户对延迟的感知。 | 交互/技术 |
| **P1 (严重)** | **教育** | 增加费曼模式的 **"Give Up" (放弃)** 阈值，防止死循环。 | 产品 |
| **P1 (严重)** | **数据** | 引入 **Vector Store** 替换单纯的 SQLite KV 存储。 | 技术 |
| **P2 (一般)** | **进化** | 为不同形态设计差异化的 **Skill Set (技能树)**。 | 产品 |
