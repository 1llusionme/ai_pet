# 2. 产品需求文档 (PRD) 与执行计划

## 2.1 概述
**产品名称**: MindShadow (暂定名)
**基础项目**: [Second-Me](https://github.com/mindverse/Second-Me)
**平台**: Web (Localhost) + 桌面通知
**技术栈**:
*   **后端**: Python (Flask), `llama.cpp` (推理), ChromaDB (向量库), SQLite (关系型数据库)。
*   **前端**: Next.js (React), TailwindCSS。
*   **新增组件**: `APScheduler` (任务调度), `Web Push API` (通知)。

## 2.2 功能需求 (MVP) 与产品总监评审意见

> **[产品总监批注] 整体基调：克制与聚焦**
> 作为一个极简的 MVP，我们当前的“什么都想做”的倾向有点危险。AI 学习伴侣的核心壁垒不在于“能解析多少种文件”，而在于**“主动触达的文案是否击中人心”**以及**“记忆是否真的懂我”**。MVP 必须砍掉所有边缘功能，将 80% 的精力投入到调度器和 Prompt 的打磨上。

### 特性 1: "影子" 身份 (系统提示词工程) - P0 (核心灵魂)
*   **需求**: AI 不能表现得像一个通用助手（“我能为您做什么？”）。它必须表现得像一个**伙伴**（“我们正在一起学习 X”）。
*   **实现**:
    *   修改 Second-Me 中的 `prompts/system_prompt.txt`。
    *   动态注入 L1 记忆（用户模式）到系统提示词中。
    *   *基调*: 鼓励性、简洁、好奇。
*   **补充/优化 (产品总监要求)**: 
    *   **人设一致性是生死线**。需定义**“过度热情限制”**，避免每次对话都强行鼓励，引入随机性或基于情绪状态的动态调整。
    *   **不要废话**：MVP 阶段的 Prompt 必须加上一条硬规则：“回答绝不超过 3 句话，不要列点，像发微信一样沟通”。

### 特性 2: 知识摄入与 "摘要" - P1 (基础支撑，需降级)
*   **需求**: 用户可以上传 PDF/Markdown 文件。AI 解析、总结，并**存储**一个“待测验”状态。
*   **用户流程**:
    1.  用户上传 `physics_notes.pdf`。
    2.  AI: “收到了。我看完了。核心概念似乎是‘量子纠缠’。明天我会考考你。”
    3.  (系统安排一个 T+24h 的任务)。
*   **补充/优化 (产品总监要求)**:
    *   **极端降维**：砍掉 PDF 解析！MVP 阶段只允许用户**“直接粘贴文本”**或**“上传纯 .txt/.md”**。不要在 PDF 解析（排版、图表、OCR）上浪费 1 分钟。我们的核心是“测验和提醒”，不是“文档阅读器”。
    *   **异常处理**: 文件过大时直接截断（例如前 5000 字），并提示：“太长不看，我先学了前半部分。”

### 特性 3: 主动调度器 (引擎) - P0 (核心差异化)
*   **需求**: 一个后台服务，检查用户状态并触发事件。
*   **逻辑**:
    *   `CheckInterval`: 每 1 小时。
    *   `Condition`: `if (LastActive > 24h) AND (PendingQuizCount > 0)`
    *   `Action`: 通过 LLM 生成通知内容 -> 推送到前端。
*   **技术**:
    *   在 Flask 应用中添加 `APScheduler`。
    *   在 SQLite 中创建 `ScheduledTask` 表来追踪待办提醒。
*   **补充/优化 (产品总监要求)**:
    *   **防打扰机制（免打扰时间 DND）**: 必须增加系统级别的免打扰时间段（例如 23:00 - 08:00）。
    *   **触达文案的“钩子”设计**：主动发出的消息绝对不能是“该学习了”。必须是**带悬念的**或**反常识的**（例如：“昨天你笔记里提到的‘熵增’，我发现个有意思的比喻，你想听吗？”）。调度器不仅要管“何时发”，更要管“发什么能让人点开”。

### 特性 4: "镜像" 仪表盘 - P2 (延后开发，MVP不需要)
*   **需求**: 可视化展示 AI 对你的“认知”。
*   **UI**:
    *   "当前关注": (例如 "学习 Python")
    *   "困难点": (例如 "递归", "列表推导式")
    *   "强项": (例如 "数据类型")
*   **来源**: 直接读取 Second-Me 的 L1/L2 记忆聚类。
*   **补充/优化 (产品总监要求)**:
    *   **直接砍掉 UI**：MVP 阶段用户不需要一个华丽的 Dashboard。AI 对用户的认知应该**直接在对话中体现**。例如 AI 突然说：“我发现你最近三次都在问循环的问题，是不是卡在这了？”。
    *   这才是真正的“伴侣感”，而不是冰冷的图表。此特性在第一版中彻底从 UI 中移除，仅保留后台的数据聚类逻辑。

## 2.3 技术架构变更

### 2.3.1 后端 (`lpm_kernel`) 修改
1.  **安装 `APScheduler`**:
    ```bash
    poetry add APScheduler
    ```
2.  **新服务: `SchedulerService`**:
    *   初始化后台调度器。
    *   定义 `jobs/daily_nudge.py`:
        *   获取最近的 L0 记忆。
        *   调用 LLM 使用 `proactive_nudge_prompt`。
        *   保存结果到 `notifications` 表。

3.  **API 扩展**:
    *   `GET /api/notifications/pending`: 前端轮询此接口显示“吐司”通知或角标。
    *   `POST /api/ingest/learning_material`: 学习资料的专用接口。

### 2.3.2 前端 (`lpm_frontend`) 修改
1.  **通知组件**:
    *   一个常驻的“铃铛”图标或“吐司”弹窗，当 `GET /api/notifications/pending` 返回数据时显示。
2.  **仪表盘小部件**:
    *   在侧边栏添加“记忆镜像”区域，可视化 L1/L2 数据。

## 2.6 风险与缓解
*   **风险**: 测验中的 LLM 幻觉。
    *   *缓解*: 强制 LLM 在提问时引用源文档（L0 记忆上下文）。并且在 UI 上提供“查看原文”按钮。
*   **风险**: 烦人/过度打扰。
    *   **缓解**: 严格的“冷却”期（例如，每 12 小时最多 1 次提醒），并严格遵守“免打扰时间 (DND)”。
*   **风险**: 用户不活跃导致“死循环”。
    *   *缓解*: 如果连续 3 次主动推送用户均无响应，系统应进入“休眠模式”，直到用户下一次主动发起交互才重新激活，避免变成垃圾信息发送机。

## 2.7 移动端演进评估 (Mobile Feasibility)

将此产品迁移至移动端（iOS/Android）是一个极具吸引力但也充满技术挑战的方向。以下是详细的可行性和难度评估：

### 1. 核心技术可行性 (可行性：中等偏上)
*   **本地 LLM 推理**: 完全可行。得益于 `llama.cpp` 的移动端适配，现在可以通过 React Native (如 `llama.rn`, `react-native-executorch` 或 `react-native-ai` 基于 MLC LLM) 在较新的手机上（如 iPhone 14/16, 高端 Android）以 10-15 tokens/sec 的速度运行 1B-3B 参数的小模型（如 Llama-3.2-1B/3B, Qwen-2.5-1.5B）。
*   **本地向量数据库**: 可行。不能直接使用 ChromaDB（基于 Python），需要替换为支持移动端的本地向量方案，例如带有 `sqlite-vec` / `sqlite-vss` 扩展的 SQLite，或者 ObjectBox。

### 2. 最大技术难点与架构冲突 (难度：极高)
当前桌面端架构依赖 Python `APScheduler` 在后台**持续运行并调用 LLM 生成通知**。在移动端，这面临操作系统级别的巨大限制：
*   **后台内存与算力限制**: iOS 和 Android 严格限制后台应用的 CPU 和内存使用。LLM 推理通常需要 1GB-3GB 内存，在应用退到后台时尝试拉起 LLM 生成“主动提醒”极大概率会被系统直接 Kill（杀死进程）。
*   **耗电量**: 在手机后台频繁运行 LLM 会导致电池飞速消耗。

### 3. 移动端架构重构建议 (两种演进路线)

**路线 A: 纯本地移动端 (100% On-Device)**
*   **技术栈**: React Native + `llama.rn` + `sqlite-vec` (本地向量搜索)。
*   **“主动机制”的妥协设计**:
    *   **前置生成**: 当用户在使用 App（前台活跃）结束时，**提前**让本地 LLM 生成好明天要发送的“提醒/测验内容”，并利用系统的**本地定时通知 (Local Notifications)** 设定在 24 小时后发送。
    *   **唤醒后生成**: 设定一个固定的本地定时通知（固定文案如“你昨天学了量子力学，来看看今天的测验？”），当用户点击通知**打开 App 后**，再即时调用本地 LLM 生成具体内容。

**路线 B: 个人云端 + 移动端前台 (Client-Server 推荐方案)**
*   **技术栈**: 保持现有的 Python 本地电脑服务端 (PC/Mac/NAS) + 开发一个轻量级 React Native App 或 PWA。
*   **逻辑**: 重计算（LLM、记忆聚类、调度器）全部留在用户的个人电脑上运行。手机端仅作为交互界面。
*   **通知机制**: 电脑端通过 Web Push API 或第三方推送服务（如 Bark, Pushover，保持数据加密）将生成的通知推送到手机。
*   **优点**: 规避了手机算力和电量瓶颈，开发难度低（复用现有架构）。

### 4. 结论与下一步建议
如果您希望 MVP 阶段就上手机：
1.  **强烈建议先采用“路线 B”**，将网页版封装为 PWA（渐进式 Web 应用）添加到手机桌面，利用电脑提供算力，跑通核心的“主动影子”产品逻辑。
2.  若坚持“路线 A”（纯本地 App），需大幅修改 PRD 中的特性 3（主动调度器），放弃 Python 后端，全面拥抱 React Native，并采用“前置生成”策略来绕过手机操作系统的后台限制。
