# MindShadow iOS 上线执行计划（V1）

## 1. 目标与上线定义

- 目标平台：iOS（App Store TestFlight 内测优先，随后正式上架）。
- 产品形态：iOS 客户端 + 现有 Python 后端（统一 API）。
- 上线定义：用户可下载安装、可登录/进入聊天、可稳定调用模型、关键错误可追踪、可热修复。

## 2. 路线选择与结论

### 方案 A：WebView 封装（最快）

- 用原有前端页面封装成 iOS 容器应用（WKWebView）。
- 优点：上线最快，复用现有 UI 和逻辑最多。
- 风险：原生体验一般，推送与系统能力接入受限。

### 方案 B：原生重构（最佳体验）

- 用 SwiftUI 重做聊天主流程，后端 API 保持不变。
- 优点：体验和性能最好，系统能力完整（通知、权限、后台）。
- 风险：研发周期更长。

### 方案 C：混合路线（推荐）

- 第一阶段先上 A 快速进 TestFlight；
- 第二阶段逐步替换核心页面为原生（先聊天，再资产页，再设置页）。
- 结论：兼顾速度和长期质量，最适合当前阶段。

## 3. 分阶段计划

### 阶段 0：上线冻结（1-2 天）

- 锁定 API 契约：`/api/chat`、`/api/history`、`/api/health`、上传相关接口。
- 明确首发范围：聊天、会话切换、基础学习资产查看。
- 暂不首发：复杂运营功能、A/B 工具页。

### 阶段 1：iOS 壳工程与接入（2-4 天）

- 新建 iOS 工程（Bundle ID、签名、环境配置）。
- 接入 WebView 容器，支持生产域名与灰度域名切换。
- 增加基础原生桥接能力：网络状态、版本号、设备信息上报。

### 阶段 2：后端生产化（并行 2-3 天）

- 生产部署后端（建议云托管 + HTTPS + 健康检查）。
- 配置生产环境变量与密钥管理（模型 Key 不入仓）。
- 启用日志聚合与错误告警，建立 API 可用性监控。

### 阶段 3：TestFlight 内测（3-5 天）

- 提交 TestFlight 构建包，邀请种子用户测试。
- 执行验收清单：登录/会话/发送消息/网络异常/模型降级。
- 收集崩溃和关键体验问题，完成一轮修复。

### 阶段 4：App Store 上架（2-4 天）

- 完成 App Store Connect 元数据：标题、副标题、截图、隐私政策。
- 完成隐私问卷与权限文案（网络、通知、相册/文件如涉及）。
- 提交审核并跟踪反馈，必要时快速修订再提审。

### 阶段 5：首发后两周运营（持续）

- 监控北极星指标：次日留存、消息发送成功率、平均会话长度。
- 每 48 小时做一次问题分级：阻断问题优先热修复。
- 开始原生化改造迭代（按混合路线逐步替换）。

## 4. 质量门槛（Go/No-Go）

- 稳定性：关键路径崩溃率 < 1%。
- 可用性：`/api/chat` 成功率 >= 99%（首发目标）。
- 性能：首屏可交互 < 2.5 秒（4G 网络）。
- 体验：发送消息到首 token 响应体感稳定，无长时间卡死。

## 5. 必备资源清单

- Apple Developer 账号（组织或个人）。
- App Store Connect 权限（管理构建和发布）。
- 后端生产环境（域名、HTTPS 证书、日志平台）。
- 模型服务密钥与额度预算。
- 隐私政策页面与服务条款页面（可先简版）。

## 6. 风险与预案

- 审核风险：内容安全、隐私声明不完整。
  - 预案：补充内容策略与举报入口，提前准备审核说明。
- 模型不稳定：高峰期响应波动。
  - 预案：降级到本地/兜底回复，前端给出明确提示。
- WebView 体验争议：
  - 预案：首发即排期“聊天页原生化”作为 P1。

## 7. 本周执行顺序（建议）

1. 完成 Apple 账号与 Bundle ID。
2. 完成后端生产部署与 HTTPS。
3. 完成 iOS 壳工程并打第一包。
4. 完成 TestFlight 内测和问题回收。
5. 完成上架素材与审核提交流程。
