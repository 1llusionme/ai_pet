# MindShadow 新增需求文档：助手输出内容质量优化（结构化可读 + 学习富文本）

**版本**: 1.0（新增需求）  
**状态**: 待评审  
**目标版本**: Web 聊天主路径增量迭代（不新增页面）  
**负责人**: 产品 + 前端 + 后端 + 算法

---

## 1. 需求背景与目标

### 1.1 背景问题
当前助手回答已具备“快”和“可执行”的特点，但在学习场景下仍有三个明显体验缺口：
1. 文本结构不稳定：同类问题在段落组织上差异大，用户扫读成本高。  
2. 重点不突出：关键概念、易错点、记忆钩子缺少视觉强调。  
3. 学习记录不友好：答案很难直接当作“可复习笔记”保存和回看。

### 1.2 本期目标
在不改变当前聊天页主结构与视觉调性的前提下，把助手回答升级为“可扫读、可标记、可复习”的学习型输出。

### 1.3 成功定义
1. 用户在 5 秒内能定位“结论、依据、行动建议”。  
2. 回答支持基础富文本并在当前消息气泡内稳定渲染。  
3. 输出风格跨题型更一致，降低“看不懂/太乱”的反馈。

### 1.4 非目标（本期不做）
1. 不做独立笔记编辑器页面。  
2. 不做复杂 WYSIWYG 富文本编辑。  
3. 不做多主题皮肤系统。  
4. 不做全量历史消息重排，仅作用于新产生回复。

---

## 2. 产品方案总览

### 2.1 功能名称
“学习表达增强（Readable Learning Output）”

### 2.2 核心能力
1. **结构化回答骨架**：统一“结论 -> 解释 -> 行动”三段式。  
2. **学习富文本渲染**：支持列表、强调、重点标注、分隔块等学习常用表达。  
3. **重点信息强化**：关键术语、口诀、易错提醒可视化呈现。  
4. **稳定降级机制**：模型未按格式输出时自动回退到普通文本显示，不影响聊天可用性。

### 2.3 适配原则（与现有 UI 一致）
1. 不新增页面，全部在现有 AI 消息气泡中完成。  
2. 沿用当前圆角、柔和渐变、轻动效风格。  
3. 保持移动端优先，单手浏览不拥挤。  
4. 不增加输入区学习负担，不新增复杂操作按钮。

---

## 3. 用户体验设计（基于当前界面）

### 3.1 页面锚点
1. 聊天消息流容器：`ChatPage.tsx`。  
2. AI 消息渲染入口：`MessageBubble.tsx` 中 `message.content` 展示区。  
3. 引用来源模块保持现有交互，不与本需求冲突。

### 3.2 输出展示规格（AI 回复区）
AI 回复默认按“学习笔记卡片文本”渲染，允许以下元素：
1. 段落标题（用于“结论 / 依据 / 下一步”）  
2. 项目符号列表（用于步骤、要点、对比）  
3. 重点强调（粗体）  
4. 重点标记（荧光笔样式高亮）  
5. 引用块（用于口诀、定义、原文摘录）

### 3.3 视觉规范（保持当前风格）
1. 标题：沿用紫灰色系，字重高于正文。  
2. 列表：紧凑行距，控制层级深度为 1 级。  
3. 高亮：采用柔和底色（黄/青二选一），避免刺眼。  
4. 引用块：圆角浅底 + 左侧强调条，避免厚重边框。  
5. 文本行长：移动端保持可读宽度，不出现超长横向滚动。

---

## 4. 内容策略与生成规则

### 4.1 新的回答结构协议（模型侧）
模型默认遵守以下结构，除非用户明确要求自由发挥：
1. **结论**：直接回答题目核心。  
2. **为什么**：给依据或记忆钩子。  
3. **怎么做**：给 1~3 条可执行动作。

### 4.2 语言质量规则
1. 句子短而完整，减少长串复合句。  
2. 专业术语出现时尽量伴随一句白话解释。  
3. 同段不混入过多信息点，保持单段单目标。  
4. 口语化但不随意，避免网络化碎片表达。

### 4.3 富文本标记约定（模型输出协议）
为降低实现复杂度，本期采用“Markdown 子集 + 自定义高亮标记”：
1. 列表：`-` 或 `1.`  
2. 强调：`**重点**`  
3. 引用：`>`  
4. 高亮：`==重点词==`

说明：若模型未输出上述标记，前端仍按普通段落渲染；若输出非法嵌套，前端执行容错清洗后渲染。

---

## 5. 前后端产品需求

### 5.1 后端需求
1. 更新聊天与视觉问答系统提示词，允许并鼓励输出结构化标记。  
2. 增加回答结构约束文案：优先三段式，必要时使用项目符号。  
3. 维持现有“证据不足要明确不确定”的可靠性约束。  
4. 保持接口字段兼容，不强制新增响应字段。

### 5.2 前端需求
1. 在 `MessageBubble` 增加安全富文本渲染器（仅支持白名单语法）。  
2. 为列表、引用、高亮提供与当前主题一致的样式 token。  
3. 兜底机制：渲染异常时回退为纯文本。  
4. 与“引用来源”卡片共存，不挤压主要阅读区域。

### 5.3 安全与稳定要求
1. 禁止渲染任意 HTML。  
2. 链接跳转必须使用安全属性。  
3. 超长回答自动分段并限制单块最大高度，避免 UI 撑爆。

---

## 6. 交互细节与边界处理

### 6.1 正常流程
1. 用户提问。  
2. 助手返回结构化文本。  
3. 消息气泡按富文本规则渲染。  
4. 用户可直接扫读重点并继续追问。

### 6.2 异常处理
1. 标记语法不闭合：自动按普通文本显示该片段。  
2. 列表层级过深：超出层级按普通段落降级。  
3. 高亮过多：单条消息最多高亮 8 处，其余转为粗体。  
4. 回答过长：保留“先给结论”在首屏可见，长内容折叠后展开。

---

## 7. 验收标准（DoD）

### 7.1 功能验收
1. AI 消息支持段落、列表、强调、引用、高亮五类元素。  
2. 高亮与引用样式在当前主题下清晰且不突兀。  
3. 富文本与“引用来源”可同时展示，无布局错乱。  
4. 不支持语法不会导致页面报错或白屏。

### 7.2 质量验收
1. 随机抽检 50 条回答，三段式结构命中率 >= 80%。  
2. 用户主观评价中“更易读”反馈占比显著提升。  
3. 首屏阅读时间下降，重复追问“你在说什么”的比例下降。

### 7.3 技术验收
1. 渲染链路无高危 XSS 风险。  
2. 移动端渲染流畅，无明显掉帧。  
3. 现有聊天、引用来源、看板功能无回归。

---

## 8. 数据指标与实验设计

### 8.1 核心指标
1. 可读性满意度（问后轻反馈）  
2. 追问质量提升率（从“解释不清”向“深入问题”迁移）  
3. 首次回答解决率（单轮结束比例）  
4. 重点内容交互率（高亮/列表段落被复制或引用占比）

### 8.2 建议埋点
1. `answer_richtext_rendered`  
2. `answer_highlight_exposed`  
3. `answer_structure_detected`  
4. `answer_render_fallback_triggered`

## 9. 迭代计划

### Phase A（本期）
1. 提示词升级为结构化输出协议。  
2. 消息气泡支持 Markdown 子集 + 高亮标记。  
3. 完成容错与回退策略。  
4. 上线基础埋点与指标看板。

### Phase B（后续）
1. 一键“转学习卡片”导出。  
2. 用户偏好设置（简洁版 / 详细版 / 速记版）。  
3. 重点术语自动收集与错题联动复盘。

---

## 10. 风险与应对

1. **风险：格式过重影响自然对话感**  
   - 应对：默认轻结构，用户可要求“简短回答模式”。  
2. **风险：模型输出格式漂移**  
   - 应对：前端容错解析 + 后端提示词强约束。  
3. **风险：视觉信息密度过高**  
   - 应对：限制高亮数量与列表层级，强调“首屏结论优先”。

---

## 11. 关键决策（供评审）

1. 高亮风格默认采用黄色还是青色？（建议黄色）  
2. 超长回答是否默认折叠？（建议超过 10 行折叠）  
3. 三段式标题是否始终显示？（建议显示，便于扫读）  
4. 是否提供“纯文本模式”开关？（建议 Phase B 再做）

## 12. Phase B 扩展需求：学习资产库与风格弹窗

### 12.1 目标
1. 把“导出的学习卡片 + 错题复盘记录”集中展示为可回看资产，降低“导出了但找不到”的挫败感。  
2. 回答风格偏好入口保持顶部按钮位，但交互升级为弹窗筛选，视觉与现有卡片化风格统一。  
3. 所有新能力在聊天页内完成，不新增独立路由页面。

### 12.2 用户问题与价值
1. 用户已开始使用“转学习卡片”，但缺少可视化沉淀区，无法形成稳定复习路径。  
2. 当前风格偏好入口可发现性和美观度不足，和产品整体“圆角卡片 + 轻动效”调性不一致。  
3. 用户希望快速切换“简洁/详细/速记”，并立即感知选择结果。

### 12.3 方案范围
1. 新增“学习资产库”弹窗（入口放在聊天页右上角工具区）。  
2. 资产库至少包含两个分区：  
   - 学习卡片（memory cards）  
   - 错题复盘（review records）  
3. 新增风格筛选弹窗（复用右上角原“灵光一闪”按钮入口，不再触发轻推）。  
4. 支持风格选项：简洁版 / 详细版 / 速记版，选择后落库到 user profile。

### 12.4 学习资产库交互要求
1. 默认展示最近资产，按时间倒序。  
2. 提供分段切换（全部 / 学习卡片 / 错题复盘），切换有轻动画反馈。  
3. 卡片条目需展示：标题（或主题）、摘要、标签、时间。  
4. 空状态需明确引导：告诉用户可通过“转学习卡片”生成首条资产。  
5. 加载态需与当前 UI 一致（骨架/呼吸感均可），避免突兀闪烁。

### 12.5 风格偏好弹窗交互要求
1. 点击右上角入口后弹出风格选择层，带出现/消失动效。  
2. 当前选中项需有明显视觉状态（底色、描边、图标或标签）。  
3. 点击选项后立即反馈保存结果（系统消息或轻提示）。  
4. 保存失败需回滚到原风格并给到明确错误提示。  
5. 不新增复杂配置项，本期仅做单选风格切换。

### 12.6 后端与接口约束
1. 学习资产库读取接口复用现有能力：  
   - `GET /api/memory-cards`  
   - `GET /api/review-records`  
2. 风格偏好写入复用：`POST /api/profile`。  
3. 不新增鉴权模型，不改现有 user_id 机制。

### 12.7 验收标准（DoD）
1. 用户能在聊天页内打开学习资产库并查看最近卡片与复盘记录。  
2. 用户能通过顶部风格弹窗切换回答风格且后端持久化成功。  
3. 右上角原“灵光一闪”按钮不再触发轻推逻辑。  
4. 弹窗视觉、圆角、色彩、动效与现有 Dashboard/Eval 弹窗保持统一。  
5. 前端构建通过、后端回归测试通过，无核心功能回归。

## 13. Phase B+ 扩展需求：图表结构可读性与学习卡片防重复导出

### 13.1 背景问题
1. 当前助手在输出“流派对比、步骤对照、概念矩阵”等图表型内容时，常用纯文本 `|` 拼接表达。  
2. 纯文本图表在不同设备宽度和消息气泡宽度下会严重换行，导致列错位、阅读断裂。  
3. “转学习卡片”按钮在成功后仍可重复点击，容易误触生成多个相似资产，增加清理成本。

### 13.2 本期目标
1. 让图表型内容在窄屏和宽屏都可读，避免“看起来像乱码表格”。  
2. “转学习卡片”在单条回答内只允许成功一次，成功后显式锁定并反馈状态。  
3. 保持现有聊天页架构不变，仅做消息渲染层和交互状态增强。

### 13.3 前端需求
1. `MessageBubble` 增加图表结构识别与渲染：支持 Markdown 表格（表头 + 分隔线 + 数据行）识别。  
2. 表格渲染采用“横向滚动容器 + 固定最小列宽”，避免列内容被强行压扁。  
3. 移动端保留视觉层级：表头高亮、行分隔清晰、单元格支持自动换行且不溢出。  
4. 识别失败时自动降级为普通段落，不阻断消息展示。  
5. “转学习卡片”按钮增加三态：可点击、处理中、已整理；“已整理”态禁用点击。  
6. 按钮成功态文案为“已整理成学习卡片”，并保持到该消息生命周期结束。

### 13.4 内容策略与输出约束
1. 当回答包含对比表时，模型优先输出 Markdown 表格，不再推荐裸文本 `|` 连写。  
2. 表格字段控制在 2~5 列，避免超宽表首屏不可读。  
3. 表格下方建议补一行结论，确保即使不看表也能理解核心信息。

### 13.5 交互与异常处理
1. 点击“转学习卡片”后立即进入处理中态，防止并发重复提交。  
2. 导出成功：切换“已整理成学习卡片”，禁用按钮并展示系统成功提示。  
3. 导出失败：恢复可点击态，并展示失败原因。  
4. 若页面刷新，按钮状态按当前会话消息状态重建，不额外引入后端字段。

### 13.6 验收标准（DoD）
1. 同一条包含 Markdown 表格的回答，在移动端与桌面端均可完整阅读，不发生列语义混乱。  
2. 表格渲染失败不会报错或白屏，自动回退为普通文本。  
3. 单条回答点击“转学习卡片”成功后不可再次点击，文案改为“已整理成学习卡片”。  
4. 导出失败路径可恢复重试，不会卡死在处理中。  
5. 前端构建通过，后端回归测试通过，无现有功能回归。

## 14. Phase B++ 扩展需求：学习资产库-学习卡片可读性与交互升级

### 14.1 背景问题
1. 学习卡片当前仅静态展示，长内容常被卡片高度截断，用户无法查看全文。  
2. 卡片正文直接使用纯文本截断，信息密度高但可读性弱，不利于快速复习。  
3. 卡片承载了“沉淀资产”的角色，但缺少“重点摘要 + 深度查看”的双层信息结构。

### 14.2 本期目标
1. 学习卡片首屏只展示“精简重点”，用户可在 3 秒内看懂这张卡的核心。  
2. 提供明确交互查看完整内容，解决被截断不可读问题。  
3. 详情阅读沿用消息气泡富文本风格，降低认知切换成本。

### 14.3 产品方案
1. 卡片列表改为“摘要卡”模式：标题 + 重点条目（最多 3 条）+ 标签 + 时间。  
2. 每张卡提供“查看全文”入口，点击后打开详情阅读层。  
3. 详情层使用富文本渲染规则（标题、列表、引用、重点高亮），与聊天消息视觉一致。  
4. 详情层支持滚动阅读，保证长内容完整可达。  
5. 若内容无法提炼重点，降级展示首段关键句，确保列表不空白。

### 14.4 内容格式策略
1. 摘要优先提取结构化信息：标题段 > 列表项 > 首段结论句。  
2. 单条重点长度控制，避免超长句；超长则省略显示。  
3. 摘要仅承载“复习导航”功能，完整论述统一在详情层承载。  
4. 卡片默认不展示大段正文，避免滚动列表时视觉噪音。

### 14.5 交互与状态要求
1. 点击“查看全文”后，详情层显示卡片标题、标签、更新时间、富文本正文。  
2. 详情层可关闭并返回原列表位置，不刷新筛选状态。  
3. 列表与详情之间切换需有轻动效，保持现有产品交互节奏。  
4. 空状态、加载态与现有资产库视觉体系保持统一。

### 14.6 验收标准（DoD）
1. 任意长度卡片均可通过详情交互查看完整正文。  
2. 列表中的学习卡片默认展示精简重点，内容可快速扫读。  
3. 详情正文支持富文本渲染，整体风格与消息气泡一致。  
4. 学习卡片筛选、错题复盘展示不受本次改造影响。  
5. 前端构建通过，无回归报错。
