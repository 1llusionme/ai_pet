# MindShadow 记忆召回升级计划（结构化优先，向量增量）

**版本**: v1.0  
**状态**: 待执行  
**负责人**: 产品负责人 + 技术联合创始人（AI）  
**适用范围**: `/api/chat`、`/api/chat/stream`、`MemoryService`、`LLMService` 相关链路

---

## 1. 目标与结论

本计划的核心结论是：**先把当前结构化记忆的召回准确率与用户体验跑到上限，再引入向量层做语义补强**。  
不是因为向量方案不好，而是因为当前系统已经具备可用记忆骨架，先做可观测和低风险优化，能更快拿到稳定收益，并避免“同时改太多导致无法归因”。

### 1.1 业务目标

1. 提高“AI 真记住我”的体感一致性，减少用户纠正 AI 的次数。  
2. 提升跨会话问题中的历史命中率（偏好、薄弱点、目标分、考试日期等）。  
3. 在不破坏现有学习闭环（计划-练习-复盘-周报-轻推）的前提下，逐步增强语义召回能力。  
4. 确保每一步都可回滚、可解释、可衡量。

### 1.2 技术策略

1. **阶段A（可观测）**：先建立召回质量观测，识别“抽取问题 / 召回问题 / 生成问题”。  
2. **阶段B（结构化强化）**：优化画像抽取与上下文拼装顺序，把现有方案做满。  
3. **阶段C（混合召回）**：保留结构化硬约束，引入向量语义召回作为补充通道。  
4. **阶段D（灰度与扩展）**：小流量验证效果与成本，达标后逐步放量。

## 3. 指标体系与达标门槛

### 3.1 北极星指标

1. **记忆命中率（Memory Hit Rate）**  
   定义：回答中正确引用用户历史信息的会话占比。  
2. **纠正率（User Correction Rate）**  
   定义：用户在下一条消息中显式纠正 AI 的占比。  
3. **跨会话命中率（Cross-session Recall Rate）**  
   定义：涉及历史上下文问题中，AI 命中相关历史信息的占比。  

### 3.2 护栏指标

1. 首字延迟与流式稳定性（不牺牲交互体验）。  
2. 错误率与限流触发率（不牺牲可用性）。  
3. 单次请求上下文长度（不引发明显成本失控）。  

### 3.3 达标逻辑

1. 若阶段B后北极星指标仍低于预期，且纠正率高位不降，进入阶段C。  
2. 若阶段C灰度期间北极星提升但护栏劣化，先优化召回策略再放量。  
3. 若阶段C未显著优于阶段B，保留结构化主路径，暂停向量全量。

## 4. 分阶段执行计划

### 4.1 阶段A：可观测先行（不改主行为）

目标：建立“可解释召回诊断面板”，先看清问题。

执行项：
1. 在 chat/chat_stream 返回中附加调试信息（仅可控开关下输出）：  
   - `memory_sources_used`: recent/profile/kb/web  
   - `profile_fields_used`: exam_goal/exam_date/response_style/weak_points  
   - `fallback_reason`: remote_fail/mock_fallback/no_relevant_memory
2. 写入匿名化召回日志表（避免存储敏感原文）。  
3. 建立基础分析报表：按会话类型、触发词、用户分群看命中与纠正。  

验收标准：
1. 任意一次“记忆错乱”反馈都能定位到抽取、召回或生成环节。  
2. 能稳定输出阶段B优化的优先级清单。

### 4.2 阶段B：结构化召回强化（低风险高收益）

目标：在不引入向量基础设施的前提下，把现有方案优化到上限。

执行项：
1. 画像抽取增强（`auto_update_profile_from_message`）：  
   - 同义表达归一（如“老忘”“总混”“记不牢”）  
   - 否定清洗（如“不是A是B”避免错写）  
   - 日期语义补全（相对日期、月日表达一致化）
2. 上下文拼装重排：  
   - 硬约束信息（考试日期/目标分/学习节奏）优先  
   - 再放 recent messages  
   - 最后补 kb/web 证据
3. 回答约束强化：  
   - 提示词中明确“仅引用可验证记忆，无法确认要说明不确定”  
   - 减少“伪记忆自信表达”
4. 失败保护：  
   - 若画像字段冲突，优先最近可信字段并打标冲突事件  
   - 不因抽取失败阻断主对话

验收标准：
1. 北极星指标相对阶段A有明确提升。  
2. 用户对“你记错了”的反馈下降。  
3. 无新增高优先级线上稳定性问题。

### 4.3 阶段C：混合记忆（结构化 + 向量）灰度

目标：只补“结构化难覆盖”的语义召回场景，不推翻现有架构。

执行项：
1. 新增语义记忆索引层：  
   - 索引对象：用户历史消息、复盘记录、记忆卡片（可配置）  
   - 颗粒度：短片段切片，保留来源与时间戳
2. 检索策略改为 Hybrid：  
   - Structured Filter（硬约束）  
   - Vector Recall（语义近邻 Top-K）  
   - Light Rerank（去重、时效、相关度）
3. 上下文融合策略：  
   - 固定预算分配：硬信息槽位 + 语义槽位  
   - 冲突时硬信息优先，语义片段作为补充证据
4. 灰度开关：  
   - 用户白名单与流量比例控制  
   - 异常自动回退结构化主路径

验收标准：
1. 跨会话命中率提升明显。  
2. 护栏指标保持在可接受范围。  
3. 单请求成本增长可控，且有收益支撑。

### 4.4 阶段D：放量与产品化

目标：将有效策略沉淀为长期能力。

执行项：
1. 固化混合召回配置中心（Top-K、时间衰减、字段权重）。  
2. 增加“记忆解释卡片”（可选）：让用户看到“这次回答参考了哪些历史”。  
3. 将召回质量纳入日常运营看板，与留存、回流、任务达标联动分析。  

验收标准：
1. 指标稳定，无放量后显著回退。  
2. 支持持续迭代而非一次性工程。

## 5. 技术设计要点（最小改造面）

### 5.1 服务改造边界

1. `MemoryService`：新增召回日志与（未来）向量检索接口。  
2. `LLMService`：新增上下文融合输入结构，明确来源标签。  
3. `app.py`：保留现有 API 契约，新增可选 debug 字段与灰度控制。  

### 5.2 数据与隐私

1. 召回日志默认不存完整原文，使用摘要与来源ID。  
2. 提供日志保留周期与清理策略。  
3. 向量层按用户隔离，避免跨用户召回污染。  

### 5.3 可回滚机制

1. 任何阶段都可通过开关回退到“仅结构化召回”。  
2. 向量检索超时或异常时，自动降级，不影响主流程回复。  
3. 灰度期间保留 A/B 对照分组，便于快速止损。

## 6. 风险清单与应对

1. **风险：召回变聪明但更慢**  
   - 应对：上下文预算控制、检索超时兜底、流式优先首字。  
2. **风险：召回增多但准确性下降**  
   - 应对：引入轻量重排与冲突检测，硬约束优先。  
3. **风险：成本上涨不可控**  
   - 应对：灰度放量、分层索引、按场景启用语义检索。  
4. **风险：调试困难**  
   - 应对：强制来源标记与召回路径日志化。

## 8. 执行顺序建议（落地版）

1. 先做阶段A，建立可观测与归因闭环。  
2. 紧接阶段B，优先做高收益低风险优化。  
3. 当阶段B收益边际下降，再进入阶段C灰度。  
4. 阶段C验证通过后，进入阶段D标准化与放量。  

这一路线确保：**每一步都有业务价值、每一步都可验证、每一步都能回退**。  
对当前 MindShadow 来说，这是“可控速度下的最快路线”。
